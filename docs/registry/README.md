# Component Registry

The registry is the family-level knowledge layer for Marwes.

It links the things that currently live in separate places:
- philosophy and usage guidance
- Figma references
- core, preset, adapter, and Storybook file trees
- semantic metadata
- AXE roadmap posture
- shared contracts and verification paths
- lightweight visual orientation without turning the registry into a screenshot archive

This is a repo-level documentation system, not a runtime package.

## Goals

For each component family, the registry should answer:
- what this family is for
- how Marwes wants it to be used
- how it is represented in Figma
- which files implement it across the repo
- which semantic attributes are canonical
- what AXE or accessibility risks remain
- which tests and docs prove the contract

## Contract split

Each family folder keeps two structured files:

- `registry.meta.json` — authored narrative metadata
- `registry.generated.json` — structural contract intended to be generator-owned
- `artifacts/component-registry.json` — repo-wide merged machine artifact

For this proof of concept, the structured registry files are generated by:
- `scripts/component-registry-sources.ts`
- `scripts/generate-component-registry.ts`
- `pnpm registry:generate`
- `pnpm registry:check`

Committed preview assets are optional.

Default preference order:
1. Storybook story references
2. Mermaid diagrams
3. small SVG previews when a family truly benefits from a static visual aid

Avoid turning the registry into a growing archive of committed PNG screenshots unless there is a strong reason.

## Folder layout

```text
registry/
├── README.md
├── schema/
│   ├── component-family.meta.schema.json
│   ├── component-family.generated.schema.json
│   └── component-registry.schema.json
├── templates/
│   ├── family.README.template.md
│   └── family.meta.template.json
└── families/
    └── button/
        ├── README.md
        ├── registry.meta.json
        ├── registry.generated.json
        ├── visuals/
        └── previews/               # optional

artifacts/
└── component-registry.json
```

## How it relates to the rest of the repo

- Architecture truth still lives in [`../reference/architecture.md`](../reference/architecture.md)
- Semantic truth still lives in `@marwes-ui/core` and [`../reference/ai-metadata.md`](../reference/ai-metadata.md)
- Cross-family accessibility support assumptions now live in [`../reference/accessibility.md`](../reference/accessibility.md)
- Accessibility strategy and support assumptions live in [`../reference/accessibility.md`](../reference/accessibility.md)
- Family accessibility execution still lives in [`../audits/README.md`](../audits/README.md)
- Compact current family blocker/manual-review status lives in [`../audits/status.md`](../audits/status.md)

The registry sits above those sources and links them together per family. Registry pages should describe family contracts and link evidence, but should not become the compact cross-family audit status table.

## Visual policy

Registry visuals are for orientation, not canonical rendering truth.

Use this order by default:
- Storybook story references for real runtime visuals
- Mermaid diagrams for structure and interaction understanding
- optional preview assets only when the family benefits materially from a static visual summary

When preview assets are needed:
- prefer SVG over PNG for committed files
- keep the count small
- avoid duplicating many state screenshots across families
