# React Native Support Plan

Branch: `react-native-evaluation`
Base branch: `add/svelte`
Status: planning artifact only
Validation pass: strengthened after reviewing current repo architecture and React Native styling/a11y
constraints

## Executive Summary

Marwes can support React Native, but it should not be treated as "React with different tags".
The web adapters are thin because the browser gives Marwes strong native semantics, CSS custom
properties, pseudo states, form controls, and Storybook a11y tooling. React Native has a different
contract: no DOM, no CSS cascade, no real CSS variables, no native checkbox/radio/select in core,
different accessibility props, and platform-specific behavior across iOS, Android, and React Native
Web.

After validating the first draft, the cleanest direction is not "build a CSS compiler and call that
the design layer". The stronger design is a Marwes Native Design Layer:

```text
firstEdition CSS + native style manifests -> native design IR -> React Native style resolver
```

In that model, the CSS compiler is only the ingestion tool. The actual product surface is the typed
native design IR and runtime resolver. That distinction matters because React Native needs explicit
slots, platform branches, interaction state, and `ViewStyle`/`TextStyle` separation that plain CSS
selectors cannot describe reliably by themselves.

The recommended path is to build a first-class package:

```text
packages/react-native
```

with a supporting generated style layer:

```text
packages/react-native/src/styles/generated
```

generated by a Marwes-owned compiler:

```text
scripts/generate-react-native-styles.ts
```

The generator should read the existing `@marwes-ui/presets` CSS plus a small authored native style
manifest per component family, then produce typed React Native style recipes. The goal is not "all CSS
works in React Native". The goal is narrower and stronger:

1. Keep `packages/presets/src/firstEdition/*.css` as the authored visual source.
2. Use native style manifests only to describe information CSS cannot express safely for RN: slots,
   platform branches, state mapping, and native-only overrides.
3. Compile the Marwes-supported CSS subset into deterministic React Native style IR.
4. Materialize the IR into React Native style arrays at runtime from `ResolvedTheme`.
5. Fail loudly when preset CSS uses a property, selector, value, or state that cannot be represented
   honestly in React Native.
6. Let `@marwes-ui/react-native` stay as thin as React, Vue, and Svelte: call core recipes, map
   accessibility, and apply generated styles.

This gives Marwes a realistic native path without duplicating an entire design language by hand.

## Current Marwes Architecture To Preserve

Marwes already has the right foundation for a native adapter:

- `@marwes-ui/core` owns component contracts, recipe logic, semantic metadata, a11y mapping decisions,
  theme resolution, color derivation, density, typography, and theme variable names.
- `@marwes-ui/presets` owns firstEdition visual language as static CSS.
- `@marwes-ui/react`, `@marwes-ui/vue`, and `@marwes-ui/svelte` are framework adapters that apply
  core render kits and keep visual styling in presets.
- Shared contracts in `tests/contracts/*` keep adapter behavior aligned.
- Registry and parity artifacts make adapter coverage auditable.

The React Native adapter should keep these principles:

- Core remains framework-agnostic and must not import React Native.
- React Native must not import React DOM, Vue, Svelte, or browser preset runtime code.
- Preset CSS remains the visual authoring surface.
- RN-specific compromises must be explicit in docs and tests.

## External Reality Check

Important React Native constraints from current docs and ecosystem:

- React Native style props are JavaScript objects, not browser CSS; names are camelCased and values are
  usually unitless density-independent pixels.
  Reference: https://reactnative.dev/docs/style
- React Native colors support common web color formats, but CSS custom properties and cascade are not
  part of the native styling model.
  Reference: https://reactnative.dev/docs/colors
- React Native layout is Flexbox-based through Yoga, but defaults differ from web CSS.
  Reference: https://reactnative.dev/docs/flexbox
- Native platform colors exist through APIs like `PlatformColor` and `DynamicColorIOS`, which may be
  useful later but should not be the first implementation surface.
  Reference: https://reactnative.dev/docs/platformcolor
- RN accessibility uses props like `accessibilityRole`, `accessibilityLabel`, `accessibilityHint`,
  and `accessibilityState`; DOM `aria-*` output from core cannot be forwarded blindly.
  Reference: https://reactnative.dev/docs/accessibility
- Existing libraries prove CSS-to-RN ideas are viable, but they come with tradeoffs:
  - NativeWind / `react-native-css-interop` can map class-like styling into RN and support a CSS-like
    workflow, but it adds a dependency model and bundler assumptions.
    References: https://www.nativewind.dev/ and https://www.nativewind.dev/docs/api/css-interop
  - Tamagui proves compiler-first universal styling can work, but adopting it would change Marwes'
    styling model rather than preserving `@marwes-ui/presets` as source.
    Reference: https://tamagui.dev/docs/intro/introduction
  - Unistyles proves runtime theming and variants can work well in RN, but it is a styling runtime,
    not a compiler from existing Marwes CSS.
    Reference: https://www.unistyl.es/

Conclusion: use existing libraries as research input and test references, not as the core dependency
for Marwes v1 RN support. A Marwes-owned compiler gives tighter guarantees and keeps the product
aligned with static CSS and typed generated artifacts.

## Validation Result

The first draft was directionally right: React Native support should be a dedicated package, should
reuse core recipes, and should investigate compiling firstEdition preset CSS rather than rewriting the
visual system by hand.

The part worth improving is the design layer boundary. CSS alone is not expressive enough to become a
clean native design layer because:

- RN needs slot-specific output: `root`, `text`, `icon`, `control`, `track`, `thumb`, and field
  helper/error slots have different valid style prop sets.
- RN has no cascade; selector order cannot remain the only composition model.
- RN state is explicit component state: `pressed`, `focused`, `disabled`, `selected`, `invalid`,
  `expanded`, and platform hover need to be passed by the component.
- RN style arrays are ordered runtime values; generated artifacts should preserve composition order
  without pretending to be CSS.
- RN accessibility is a native prop contract, not a DOM ARIA contract.
- Some CSS is intentionally browser-only and should never enter RN output: `cursor`, `user-select`,
  `transition`, `outline-offset`, and some selector shapes.

The improved recommendation is:

1. Keep preset CSS as the visual source.
2. Add small native style manifests to describe native-only structure.
3. Generate a typed native design IR, not final flat styles only.
4. Resolve the IR at runtime against `ResolvedTheme`, component state, and platform.
5. Let RN components consume style slots, not CSS class strings.

This is a cleaner design-system solution because it preserves the web source of truth while making the
native boundary explicit and auditable.

## Non-Negotiable Product Boundaries

React Native support should be considered real only when these are true:

1. A user can install `@marwes-ui/react-native` into an Expo or bare RN app.
2. A user can wrap their app in `MarwesProvider` and render supported Marwes components.
3. Components use `@marwes-ui/core` recipes where those recipes are platform-neutral.
4. Visuals are generated from `@marwes-ui/presets/firstEdition` or from a documented native override.
5. The package has a published component support matrix, not vague parity claims.
6. Unsupported web semantics are blocked, omitted, or explicitly re-modeled for RN.
7. CI can prove generated native styles are up to date.
8. CI can prove the package typechecks and works in at least one Expo playground.

## Native Design Layer Architecture

The React Native design layer should have five explicit layers.

### Layer 1: Core Recipes

Source:

```text
packages/core/src/components/**/*
```

Responsibilities:

- component option normalization
- semantic intent
- accessibility intent where platform-neutral
- state-independent metadata
- class names for existing web adapters

Important correction: the RN adapter should not depend on parsing `kit.className` forever. That is a
reasonable bridge for a spike, but the clean model is to add platform-neutral recipe metadata over
time:

```ts
type RenderKitStyleMeta = {
  family: "button"
  slots: {
    root: ["mw-btn", "mw-btn--primary", "mw-btn--md"]
    label: ["mw-btn__label"]
    icon: ["mw-btn__icon"]
  }
  state: {
    disabled?: boolean
    invalid?: boolean
  }
}
```

This metadata can be added gradually without changing the existing web adapter behavior.

### Layer 2: Preset CSS

Source:

```text
packages/presets/src/firstEdition/*.css
packages/presets/src/firstEdition/molecules/*.css
```

Responsibilities:

- remain the visual language source for firstEdition
- keep web CSS behavior intact
- continue to support React/Vue/Svelte through static CSS

The RN generator treats this as an input, not as a runtime dependency.

### Layer 3: Native Style Manifests

Source:

```text
packages/react-native/src/styles/manifests/first-edition/*.native-style.json
```

Purpose:

CSS cannot tell the compiler everything RN needs to know. The manifest fills only that gap.

Manifest responsibilities:

- map selectors to native slots
- classify each slot as `view`, `text`, `image`, or `svg`
- declare supported states for the family
- declare platform branches where needed
- declare native-only overrides when CSS uses browser-only behavior
- mark intentionally ignored CSS declarations with a reason

Example:

```json
{
  "family": "button",
  "sourceCss": "packages/presets/src/firstEdition/button.css",
  "slots": {
    "root": {
      "type": "view",
      "selectors": [".mw-btn"]
    },
    "label": {
      "type": "text",
      "selectors": [".mw-btn__label"]
    },
    "icon": {
      "type": "view",
      "selectors": [".mw-btn__icon"]
    }
  },
  "variants": {
    "primary": [".mw-btn--primary"],
    "secondary": [".mw-btn--secondary"],
    "text": [".mw-btn--text"],
    "success": [".mw-btn--success"]
  },
  "states": {
    "pressed": [":active"],
    "focused": [":focus-visible"],
    "disabled": [":disabled"],
    "invalid": ["[data-error=\"true\"]", ".mw-btn--error"]
  },
  "ignoredDeclarations": {
    "cursor": "RN Pressable owns touch behavior.",
    "user-select": "Not a native style prop.",
    "transition": "Animation requires an explicit native animation strategy."
  }
}
```

Manifests should be small. If they become large style files, the design layer is drifting toward a
second native preset and should be challenged.

### Layer 4: Generated Native Design IR

Generated source:

```text
packages/react-native/src/styles/generated/first-edition.ir.json
packages/react-native/src/styles/generated/first-edition.ts
packages/react-native/src/styles/generated/first-edition.types.ts
```

The IR should preserve design intent, not just flatten everything into untyped objects.

Recommended IR shape:

```ts
type NativeDesignIR = {
  version: 1
  preset: "firstEdition"
  families: Record<
    string,
    {
      sourceCss: string
      slots: Record<string, NativeSlotIR>
      variants: Record<string, NativeSelectorRef[]>
      states: Record<string, NativeSelectorRef[]>
      rules: NativeRuleIR[]
      diagnostics: NativeCompilerDiagnostic[]
    }
  >
}

type NativeRuleIR = {
  selector: string
  slot: string
  mode?: "light" | "dark"
  platform?: "ios" | "android" | "web"
  state?: "default" | "pressed" | "focused" | "disabled" | "hovered" | "invalid"
  declarations: NativeDeclarationIR[]
  source: {
    file: string
    line: number
  }
}

type NativeDeclarationIR =
  | { kind: "static"; prop: string; value: string | number }
  | { kind: "token"; prop: string; token: "--mw-color-primary-base" }
  | { kind: "colorMix"; prop: string; colorToken: "--mw-color-text"; amount: number }
```

Why IR matters:

- compiler tests can assert stable intent
- runtime can materialize theme-dependent values safely
- documentation can show exactly what CSS did or did not compile
- future RN Web / iOS / Android branches can reuse the same generated model

### Layer 5: Runtime Materializer

Source:

```text
packages/react-native/src/styles/resolve-native-style.ts
```

Responsibilities:

- receive family, style metadata, theme, platform, and component state
- resolve token references from `ResolvedTheme`
- evaluate supported `color-mix` expressions
- merge style arrays in deterministic order
- return slot-specific RN style arrays

Example output:

```ts
type NativeStyleOutput = {
  root: StyleProp<ViewStyle>
  label: StyleProp<TextStyle>
  icon: StyleProp<ViewStyle>
}
```

This keeps the RN component thin:

```tsx
const kit = createButtonRecipe(props)
const state = { pressed, focused, disabled: kit.a11y.disabled }
const styles = useMarwesNativeStyles("button", kit.styleMeta, state)

return (
  <Pressable style={styles.root}>
    <Text style={styles.label}>{children}</Text>
  </Pressable>
)
```

## Compiler: CSS To Native Design IR

### Goal

Compile firstEdition preset CSS plus native style manifests into a typed native design IR.

Input:

```text
packages/presets/src/firstEdition/button.css
packages/presets/src/firstEdition/badge.css
packages/presets/src/firstEdition/molecules/input-field.css
packages/react-native/src/styles/manifests/first-edition/button.native-style.json
...
```

Output:

```text
packages/react-native/src/styles/generated/first-edition.ir.json
packages/react-native/src/styles/generated/first-edition.ts
```

Generated TypeScript facade shape:

```ts
export const firstEditionNativeStyles = {
  button: {
    sourceCss: "packages/presets/src/firstEdition/button.css",
    slots: {
      root: { type: "view" },
      label: { type: "text" },
      icon: { type: "view" }
    },
    variants: {
      primary: ["rule.button.base", "rule.button.primary"],
      secondary: ["rule.button.base", "rule.button.secondary"],
      text: ["rule.button.base", "rule.button.text"]
    },
    sizes: {
      xs: ["rule.button.size.xs"],
      sm: ["rule.button.size.sm"],
      md: ["rule.button.size.md"],
      lg: ["rule.button.size.lg"]
    },
    states: {
      disabled: ["rule.button.disabled"],
      pressed: ["rule.button.pressed"],
      focused: ["rule.button.focused"]
    }
  }
} as const
```

Runtime resolver shape:

```ts
const styles = resolveNativeStyle({
  family: "button",
  styleMeta: kit.styleMeta,
  dataAttributes: kit.dataAttributes,
  theme,
  state: { pressed, focused, disabled },
})
```

### Compiler Pipeline

1. Parse CSS with a real parser.
   - Preferred: PostCSS or Lightning CSS.
   - Avoid ad hoc regex parsing.
   - The repo already has a PostCSS override for security, so using PostCSS as a dev-only parser is
     acceptable if kept pinned and tested.

2. Normalize selectors into a Marwes RN selector model.
   - Supported:
     - `.mw-btn`
     - `.mw-btn--primary`
     - `.mw-btn.mw-btn--text`
     - `.mw-btn[data-error="true"]`
     - `.mw-theme--dark .mw-btn--primary`
     - internal slots like `.mw-toast__icon`
   - Selector-to-slot mapping must come from the native style manifest, not from naming heuristics
     alone.
   - Supported as RN state mappings:
     - `:disabled` -> `state.disabled`
     - `:active` -> `state.pressed`
     - `:focus-visible` -> `state.focused`
     - `:hover` -> optional `state.hovered` only for RN Web or pointer-capable platforms
   - Block initially:
     - arbitrary sibling selectors
     - complex combinators
     - pseudo-elements
     - layout selectors requiring cascade behavior
     - media queries unless explicitly added later

3. Normalize declarations.
   - `background` -> `backgroundColor` when value is a color.
   - `border-color` -> `borderColor`.
   - `border-radius` -> `borderRadius`.
   - `font-family` -> `fontFamily`.
   - `font-size` -> `fontSize`.
   - `font-weight` -> `fontWeight`.
   - `line-height` -> `lineHeight`.
   - `padding`, `padding-x`, `padding-y` shorthands -> RN edge props.
   - `gap` -> `gap` only after confirming RN floor supports it; otherwise expand through component
     layout wrappers.
   - `outline` / `outline-offset` -> native focus ring style where supported, otherwise a wrapper
     border/shadow model.
   - validate declaration destination: text-only props cannot be emitted into `ViewStyle`, and
     layout-only props cannot be emitted into `TextStyle` unless RN supports them.

4. Resolve token references.
   - `var(--mw-color-primary-base)` becomes a generated token reference, not a string containing
     `var(...)`.
   - Runtime resolves token reference from `ResolvedTheme`.
   - `--mw-density-*`, typography, font, radius, and status tokens map the same way.

5. Resolve unit values.
   - `px` -> number.
   - `rem` -> number using Marwes configured base, initially `16`.
   - unitless number -> number.
   - percentages only where RN accepts them.

6. Resolve color functions.
   - `color-mix(...)` is the main blocker in current presets.
   - v1 options:
     - Preferred: compile simple `color-mix(in srgb, var(--mw-color-X) N%, transparent)` into a
       runtime helper that outputs an rgba/hex value from theme tokens.
     - Alternative: move such cases into explicit theme tokens before compiling.
   - The compiler must fail on unsupported color functions.

7. Attach diagnostics and source maps.
   - every generated rule should know the source CSS file and line
   - ignored declarations require a manifest reason
   - unsupported declarations fail check mode
   - unsupported selectors fail check mode

8. Generate typed style recipes.
   - Generated code should be deterministic and committed.
   - Add `native-styles:generate` and `native-styles:check` scripts.
   - Add compiler tests using a small fixture CSS file and real firstEdition CSS.

### Why Not Just Use NativeWind?

NativeWind is valuable to evaluate, especially because it has `react-native-css-interop`, but it is
not the best default for Marwes v1:

- Marwes already has a controlled preset CSS language, not arbitrary app CSS.
- Marwes wants package-level generated artifacts and CI drift checks.
- Marwes adapters should remain thin and predictable.
- NativeWind introduces runtime/bundler assumptions that may be fine for apps but less ideal for a
  published design-system adapter.

Recommended decision:

- Build the Marwes compiler as the source of truth.
- Create a short spike comparing NativeWind as an optional interop path.
- Do not make NativeWind a dependency of `@marwes-ui/react-native` unless the spike proves a strong
  reason.

## Theme Runtime For React Native

The web provider emits CSS variables. RN provider should emit a resolved native theme object.

Provider responsibilities:

- call `resolveThemeInput` from `@marwes-ui/core`
- subscribe to `Appearance` for system light/dark when `preference="system"`
- store preference through a pluggable storage adapter, not direct AsyncStorage dependency
- expose:
  - `theme: ResolvedTheme`
  - `mode`
  - `preference`
  - `setMode`
  - `setPreference`
  - `toggleMode`
  - `nativeTokens`

Suggested API:

```tsx
import { MarwesNativeProvider, Button } from "@marwes-ui/react-native"

export function App() {
  return (
    <MarwesNativeProvider>
      <Button variant="primary">Save</Button>
    </MarwesNativeProvider>
  )
}
```

Storage should be an injected interface:

```ts
export type NativeThemePreferenceStorage = {
  read(): Promise<ThemePreference | undefined> | ThemePreference | undefined
  write(preference: ThemePreference): Promise<void> | void
}
```

Do not depend on `@react-native-async-storage/async-storage` in the first package release. Consumers
can pass it in.

## Accessibility Mapping

React Native cannot reuse DOM a11y props directly.

Add a core-to-native mapper:

```text
packages/react-native/src/accessibility/map-accessibility.ts
```

Mapping examples:

| Core/Web concept | React Native mapping |
| --- | --- |
| `aria-label` | `accessibilityLabel` |
| `aria-disabled` / `disabled` | `accessibilityState.disabled` and component-specific `disabled` |
| `aria-busy` | `accessibilityState.busy` where supported |
| `aria-expanded` | `accessibilityState.expanded` |
| `aria-checked` | `accessibilityState.checked` |
| `role="button"` | `accessibilityRole="button"` |
| `role="link"` | `accessibilityRole="link"` |
| `aria-invalid` | `accessibilityState.invalid` if supported; otherwise use `accessibilityHint` and error text |
| `aria-describedby` | no direct equivalent; compose helper/error into labels or hints carefully |
| `aria-live` | platform-specific; use announcements only for high-value cases |

Do not claim parity for field semantics until screen-reader behavior is manually checked on iOS and
Android.

| Card | `View` | High | Layout/style only. |
| Divider | `View` | High | Needs orientation support. |
| Heading | `Text` | High | Maps to styled `Text`; native heading semantics are weaker. |
| Paragraph | `Text` | High | Straightforward. |
| Spacing | `View` | High | Straightforward spacer/view utility. |
| Spinner | `ActivityIndicator`, `View` | High | Use RN primitive where possible. |
| StatTile | `View`, `Text` | High | Mostly composed visuals. |
| Avatar | `Image`, `Text`, `View` | Medium | Image loading and fallback initials. |
| Icon | `react-native-svg` | Medium | Needs optional peer or separate icon renderer. |

### Tier 2: Form Controls With RN-Specific Contracts

These need careful design because RN does not provide browser form semantics.

| Family | RN primitive | v1 feasibility | Notes |
| --- | --- | --- | --- |
| Input | `TextInput` | High | Text input maps well; field descriptions need a11y strategy. |
| Textarea | `TextInput multiline` | High | Straightforward but sizing differs. |
| Switch | `Switch` | High | RN has a native primitive. |
| Checkbox | `Pressable` + state | Medium | RN core has no checkbox; must implement role/state manually. |
| Radio | `Pressable` + state | Medium | Group behavior must be explicit. |
| Slider | external slider | Medium/Low | RN core has no slider; likely optional peer. |
| Select / Dropdown | custom modal/list | Low | Needs a native picker design; not a simple CSS translation. |
| DatePicker | external/native module | Low | Platform-specific; likely out of v1. |

### Tier 3: Defer Until Native Semantics Are Designed

| Family | Reason |
| --- | --- |
| Dialog / Modal | RN has `Modal`, but focus, back button, escape, and screen-reader behavior need a native spec. |
| Tooltip | Mobile tooltip semantics differ heavily; likely not a v1 primitive. |
| RichText | Web editor model does not map cleanly to RN without a dedicated editor dependency. |
| Tab | Possible, but should align with native navigation patterns and accessibility review. |
| Accordion | Possible with `Pressable` + state, but screen-reader expectations need validation. |
| Skeleton | Visually easy, but animation strategy and reduced motion need a platform decision. |
| Toast | Feasible, but provider/portal/layering needs platform-specific behavior. |

## API Design Direction

### Naming

Use native package naming and preserve familiar component names:

```ts
import { Button, InputField, MarwesNativeProvider } from "@marwes-ui/react-native"
```

Provider name should probably be `MarwesNativeProvider`, not `MarwesProvider`, to avoid confusion in
shared React apps. Re-exporting `MarwesProvider` as an alias can be considered later.

### Navigation Buttons

The web `Button` can render `<a>` when `href` exists. RN has no anchor equivalent in core.

Initial plan:

- `Button` supports action buttons only.
- `LinkButton` accepts `onPress` and optional `href` only for React Native Web.
- Add `to` / `href` adapters only after deciding whether to support React Navigation, Expo Router,
  or app-owned navigation callbacks.

Avoid making `@marwes-ui/react-native` depend on any navigation framework.

### Styling Escape Hatches

RN users expect `style` props.

Recommended props:

```ts
type ButtonProps = ButtonOptions & {
  children?: React.ReactNode
  onPress?: PressableProps["onPress"]
  style?: StyleProp<ViewStyle>
  textStyle?: StyleProp<TextStyle>
}
```

Keep style props slot-specific where needed:

- `style`
- `textStyle`
- `iconStyle`
- `containerStyle`
- field slot styles for molecules

Do not expose className as the primary API.

## Compiler Contract In Detail

The compiler contract is the part that should make implementation feel safe. It must define exactly
what the generator accepts, what it produces, how unsupported web CSS is handled, and what CI can
prove. The target is not "React Native CSS". The target is a native style contract derived from
Marwes preset CSS.

```text
Preset CSS + native style manifest + core style metadata
-> validated native design IR
-> typed resolver input
-> RN StyleProp output per slot
```

### Contract Boundaries

| Boundary | Owner | Contract | Must not do |
| --- | --- | --- | --- |
| Preset CSS | `@marwes-ui/presets` | Visual source for firstEdition web styles | Import React Native or add native-only syntax |
| Native manifest | `@marwes-ui/react-native` | Declares slots, state mappings, ignored declarations, native overrides | Become a second full style source |
| Core recipe metadata | `@marwes-ui/core` | Declares family, slots, variants, size, state identity | Import RN or know about native style props |
| Compiler | `scripts/generate-react-native-styles.ts` | Converts supported CSS subset into native design IR | Parse arbitrary browser CSS at runtime |
| Generated IR | `packages/react-native/src/styles/generated` | Auditable source of native style intent | Hide unsupported CSS or drop declarations silently |
| Runtime resolver | `@marwes-ui/react-native` | Materializes IR with theme, platform, and interaction state | Read CSS files, parse selectors, or mutate theme |

### Input Contract

Each compiled family must have exactly three inputs:

```text
1. preset CSS file
2. native style manifest
3. core recipe style metadata shape
```

For Button that means:

```text
packages/presets/src/firstEdition/button.css
packages/react-native/src/styles/manifests/first-edition/button.native-style.json
packages/core/src/components/atoms/button/button-recipe.ts
```

The compiler should fail if:

- the manifest points to a CSS file that does not exist
- a manifest selector is not present in the CSS source
- a CSS selector matches no manifest slot and is not explicitly ignored
- a manifest slot has no corresponding `RenderKitStyleMeta` slot
- a variant or state exists in core metadata but has no manifest mapping when it affects visuals
- generated output would change in `--check` mode

### Manifest Schema Contract

The manifest is not an escape hatch for writing native styles by hand. It is a declarative bridge
for information CSS cannot safely express for RN.

Recommended schema:

```ts
type NativeStyleManifest = {
  version: 1
  preset: "firstEdition"
  family: ComponentFamily
  sourceCss: string
  slots: Record<string, NativeSlotManifest>
  variants?: Record<string, NativeSelectorList>
  sizes?: Record<string, NativeSelectorList>
  states?: Record<string, NativeStateMapping>
  platforms?: Partial<Record<"ios" | "android" | "web", NativePlatformMapping>>
  ignoredSelectors?: Record<string, string>
  ignoredDeclarations?: Record<string, string>
  nativeOverrides?: NativeOverride[]
}

type NativeSlotManifest = {
  type: "view" | "text" | "image" | "svg"
  selectors: NativeSelectorList
  required?: boolean
}

type NativeStateMapping = {
  selectors: NativeSelectorList
  sourceState: "disabled" | "pressed" | "focused" | "hovered" | "invalid" | "selected"
}

type NativeOverride = {
  reason: string
  slot: string
  state?: string
  platform?: "ios" | "android" | "web"
  declarations: Record<string, string | number>
}

type NativePlatformMapping = {
  selectors?: NativeSelectorList
  overrides?: NativeOverride[]
}

type NativeSelectorList = string[]
```

Manifest rules:

- `version` is required so generated IR can evolve without ambiguity.
- `sourceCss` must be repo-relative.
- every slot must declare its RN style target type.
- every ignored selector/declaration must include a human-readable reason.
- `nativeOverrides` must always include a reason and should stay rare.
- if a manifest grows larger than the component CSS, stop and redesign the mapping.

### Selector Contract

Selectors are not general browser selectors. They are a constrained Marwes selector language.

Allowed in Phase 1:

| CSS selector pattern | Native interpretation |
| --- | --- |
| `.mw-btn` | base slot rule |
| `.mw-btn--primary` | variant rule |
| `.mw-btn.mw-btn--primary` | base plus variant condition |
| `.mw-btn__label` | named child slot |
| `.mw-btn[data-error="true"]` | state/data condition |
| `.mw-theme--dark .mw-btn` | theme mode branch |
| `.mw-btn:disabled` | disabled state branch |
| `.mw-btn:active` | pressed state branch |
| `.mw-btn:focus-visible` | focused state branch |
| `.mw-btn:hover` | hovered state only for RN Web or pointer-capable platforms |

Blocked until explicitly designed:

- sibling selectors
- parent selectors
- pseudo-elements
- arbitrary descendants not mapped to slots
- `@media` queries
- CSS grid selectors or layout assumptions
- selectors whose result depends on DOM cascade structure RN cannot model

Ordering rule:

```text
base -> size -> variant -> mode -> platform -> state -> native override -> user style prop
```

The compiler should encode this order in the IR. The runtime resolver should not recalculate CSS
specificity. If two compiled rules conflict at the same layer, the compiler should emit a diagnostic
that includes both source locations.

### Declaration Contract

Every CSS declaration must land in one of four buckets:

| Bucket | Meaning | CI behavior |
| --- | --- | --- |
| `supported` | maps to RN style prop or token expression | generated into IR |
| `stateMapped` | represented through Pressable/component state | generated as state rule |
| `ignored` | browser-only and intentionally irrelevant | allowed only with manifest reason |
| `unsupported` | cannot be represented honestly | fails generation/check |

Examples:

```text
background-color -> supported: backgroundColor
border-radius -> supported: borderRadius
color -> supported only for text/svg slots
cursor -> ignored with manifest reason
transition -> ignored until native animation policy exists
outline -> stateMapped or unsupported depending on component
box-shadow -> unsupported until shadow policy exists
```

Slot destination rules:

- `TextStyle` slots may receive typography and text color props.
- `ViewStyle` slots may receive layout, spacing, border, opacity, and background props.
- `ImageStyle` slots may receive dimensions, border radius, opacity, and resize-related props.
- text-only declarations must not be emitted into view slots.
- view-only layout declarations must not be emitted into text slots unless RN explicitly supports
  them and tests prove it.

### Token And Theme Contract

The generated IR must never contain unresolved `var(...)` strings.

Allowed token outputs:

```ts
type NativeDeclarationIR =
  | { kind: "static"; prop: string; value: string | number }
  | { kind: "token"; prop: string; token: ThemeTokenName }
  | { kind: "semanticToken"; prop: string; path: string[] }
  | { kind: "colorMix"; prop: string; input: NativeColorMixIR }

type ThemeTokenName = `--mw-${string}`

type NativeColorMixIR = {
  space: "srgb"
  base: { kind: "token"; token: ThemeTokenName } | { kind: "static"; value: string }
  baseAmount: number
  mix:
    | { kind: "transparent" }
    | { kind: "token"; token: ThemeTokenName }
    | { kind: "static"; value: string }
  mixAmount: number
}
```

Rules:

- CSS variables must map to known Marwes theme tokens.
- unknown CSS variables fail compilation.
- theme-mode branches must map to `mode: "light" | "dark"`.
- color functions must either compile into a typed `colorMix` IR node or fail.
- runtime token resolution must use `ResolvedTheme`, not CSS custom properties.
- generated files should include a token inventory for each family so CI can detect unknown or
  unused token references.

### Diagnostics Contract

Diagnostics should be stable and testable. Use error codes instead of only prose.

Recommended codes:

| Code | Severity | Meaning |
| --- | --- | --- |
| `RN001_UNMAPPED_SELECTOR` | error | CSS selector has no slot/state/variant mapping |
| `RN002_UNSUPPORTED_SELECTOR` | error | selector shape is outside supported subset |
| `RN003_UNSUPPORTED_DECLARATION` | error | CSS prop or value cannot compile |
| `RN004_MISSING_IGNORE_REASON` | error | ignored prop/selector has no manifest reason |
| `RN005_SLOT_TYPE_MISMATCH` | error | declaration cannot be emitted to slot style type |
| `RN006_UNKNOWN_TOKEN` | error | CSS variable does not map to a theme token |
| `RN007_UNSUPPORTED_COLOR_FUNCTION` | error | color function cannot be represented |
| `RN008_CONFLICTING_RULE_ORDER` | error | two rules conflict at same precedence layer |
| `RN009_NATIVE_OVERRIDE_TOO_BROAD` | warning | override is not scoped to a slot/state/platform |
| `RN010_MANIFEST_DRIFT` | error | manifest references missing CSS/core metadata |

Diagnostic shape:

```ts
type NativeCompilerDiagnostic = {
  code: string
  severity: "error" | "warning"
  message: string
  family: string
  selector?: string
  declaration?: string
  source?: {
    file: string
    line: number
    column?: number
  }
  manifestPath?: string
  suggestion?: string
}
```

`native-styles:check` must fail on any error diagnostic. Warnings are allowed only during the spike;
before alpha, warning policy should be tightened so support status cannot hide known drift.

### Output Contract

Generated output must be deterministic and reviewable:

```text
packages/react-native/src/styles/generated/first-edition.ir.json
packages/react-native/src/styles/generated/first-edition.ts
packages/react-native/src/styles/generated/first-edition.types.ts
packages/react-native/src/styles/generated/first-edition.diagnostics.json
```

Required generated metadata:

- generator version
- source CSS file and hash
- manifest file and hash
- generated timestamp should be avoided or omitted for stable diffs
- component family
- slot inventory
- variant inventory
- state inventory
- token inventory
- ignored declaration inventory with reasons
- diagnostics

The TypeScript facade should expose only typed data and helper accessors. RN components should not
read the JSON directly except through the registry.

### First Vertical Slice Contract

The first implementation PR should prove the full chain for Button and nothing broader:

```text
button.css
button.native-style.json
button recipe styleMeta
generated Button IR
resolveNativeStyle("button")
native Button component
Expo Button screen
```

Required behavior:

- Button uses generated styles for `root` and `label`.
- `variant`, `size`, `disabled`, `pressed`, `focused`, and `mode` all affect styles through IR.
- user `style` and `labelStyle` apply after generated styles.
- source CSS line numbers are visible in diagnostics.
- native overrides, if any, are scoped to a slot and include a reason.
- no runtime code reads CSS files.
- no runtime code parses class names after `styleMeta` exists.
- no unsupported preset CSS is silently dropped.

This is the confidence point. If Button needs large native overrides, the compiler approach should be
reassessed before adding Badge, Card, Input, or other families.

### Go / No-Go Matrix For The Compiler

The Button spike should end with an explicit decision.

Continue with compiler-first native design layer if:

- Button compiles with zero error diagnostics.
- ignored declarations are browser-only and clearly justified.
- native overrides are small, slot-scoped, and explain native behavior rather than replacing the
  component style.
- generated IR is understandable in code review.
- Expo renders Button correctly in light and dark mode.
- adding `styleMeta` to core does not make web adapters more complex.

Revise the compiler/manifest model if:

- selector mapping is correct but manifests become too verbose.
- state mapping is the hard part, but base styles compile cleanly.
- `color-mix` or token resolution needs a better shared helper.
- RN Web requires platform branches earlier than expected.

Stop the compiler-first approach and choose handwritten native styles if:

- a large share of Button declarations need native overrides.
- the manifest starts duplicating CSS values instead of describing structure.
- generated IR is harder to review than handwritten styles.
- the runtime resolver needs browser-like cascade or specificity logic.
- core metadata changes create churn for React, Vue, or Svelte.

This keeps the first spike honest. The goal is a cleaner design layer, not proving the compiler idea
at all costs.

## Native Style Quality Bar

The native design layer should be judged by these rules:

1. No untyped style blobs in component files.
   - Component files can pass `style` overrides through, but firstEdition component styles come from
     generated artifacts.

2. No runtime CSS parser.
   - All CSS parsing happens in generation scripts.

3. No `className` dependency in the final architecture.
   - The spike may parse `kit.className`, but the target state is `kit.styleMeta`.

4. No silent CSS drops.
   - Ignored declarations require a manifest reason.
   - Unsupported declarations fail `native-styles:check`.

5. No fake parity.
   - RN support matrix is separate from web parity.
   - Components are marked `experimental`, `supported`, `deferred`, or `unsupported`.

6. No native module lock-in for v1 basics.
   - Mandatory package should work in Expo without additional native installs for Tier 1 components.

7. Style output remains inspectable.
   - Generated IR is committed.
   - Generated TypeScript facade is readable.
   - Source CSS file and line are available for diagnostics.

### Generated Style Resolver

The resolver should accept platform-neutral style metadata and native state:

```ts
type NativeStyleInput = {
  family: ComponentFamily
  styleMeta: RenderKitStyleMeta
  dataAttributes?: Record<string, string | undefined>
  theme: ResolvedTheme
  platform: "ios" | "android" | "web"
  state?: {
    disabled?: boolean
    pressed?: boolean
    focused?: boolean
    hovered?: boolean
    error?: boolean
    selected?: boolean
  }
}
```

It should return slot styles:

```ts
type NativeStyleOutput = {
  root?: StyleProp<ViewStyle>
  label?: StyleProp<TextStyle>
  icon?: StyleProp<ViewStyle | TextStyle>
  control?: StyleProp<ViewStyle>
  helperText?: StyleProp<TextStyle>
  errorText?: StyleProp<TextStyle>
}
```

Slot support is important. CSS has descendants like `__icon`, `__label`, `__control`; RN needs those
as separate style objects.

## Core Changes To Consider

Keep core browser-agnostic, but add platform-neutral metadata if needed:

1. Add optional `family` and `slots` to RenderKit types.
   - Example: `family: "button"`, `slots: { root: ["mw-btn"], label: ["mw-btn__label"] }`
   - This avoids parsing `className` strings forever.
   - It also helps Vue/Svelte/React later if slot styling grows.

2. Add platform-neutral accessibility intent objects.
   - Current recipes output DOM-oriented a11y props.
   - Native adapter can start with a mapper, but long-term core may expose `accessibilityIntent`
     alongside DOM props.

3. Add `themeToNativeTokens(theme)` in core or RN package.
   - Prefer RN package first to avoid leaking native concerns into core.
   - Promote to core only if multiple non-CSS adapters need it.

1. Add native style manifest schema and compiler fixture tests.
2. Generate Button native design IR from `button.css` plus `button.native-style.json`.
3. Add `RenderKitStyleMeta` to the Button recipe, without changing web behavior.
4. Build `MarwesNativeProvider`.
5. Build native `Button`.
6. Add Expo playground with a single Button screen.
7. Expand the design layer to Badge/Card/Divider/Text.
8. Build Tier 1 package.
9. Add registry/parity support.
10. Add form-family RFC before implementation.

This order keeps risk front-loaded. If CSS compilation fails as a strategy, we find out before
building a broad adapter.

## Major Risks

### Risk: Preset CSS Is Too Browser-Shaped For Broad Compilation

The firstEdition preset contains enough browser-specific CSS that broad conversion could turn into a
second browser engine. `:has(...)`, pseudo-elements, media queries, grid, animation, transitions,
focus outlines, and `calc(...)` cannot all be treated as simple RN style props.

Mitigation:

- do a preset audit before implementing more than Button
- compile only a documented Marwes CSS subset
- keep hard families deferred until each has a native behavior spec
- define family-level support status from real compiler results, not from web parity

### Risk: Token Aliases Become Runtime CSS Variables In Disguise

Many preset values are component-local CSS variables like `--mw-button-*` or `--mw-switch-*`, which
then point at semantic tokens. If the compiler only preserves those names, RN runtime may recreate a
CSS custom-property system without browser guarantees.

Mitigation:

- resolve component-local aliases during generation
- emit semantic token references where possible
- include token inventory in generated output
- fail unknown or circular aliases

### Risk: `calc(...)` And Unit Math Hide Layout Decisions

Several preset rules use `calc(...)` for spacing, offsets, overlap, and responsive widths. RN can
accept numbers and some percentage strings, but it does not evaluate general CSS math as layout CSS.

Mitigation:

- support only numeric/token arithmetic that can be evaluated at generation time
- support runtime arithmetic only when all operands are theme tokens
- block viewport math like `100vw` until a native layout strategy exists
- require native overrides for layout formulas that depend on component structure

### Risk: Focus, Hover, Pressed, And Disabled States Do Not Share One Model

Web CSS has `:hover`, `:active`, `:focus-visible`, and `:disabled`. RN has `Pressable` state,
accessibility state, platform focus behavior, and RN Web pointer behavior. Treating these as the
same state model would overclaim parity.

Mitigation:

- map every state through component-owned state, not raw selectors
- keep `hovered` optional and platform-scoped
- test focused behavior separately for native and RN Web
- document where native focus rings differ from web focus rings

### Risk: Shadows And Elevation Are Version/Platform Dependent

Modern RN has broader shadow APIs, including `boxShadow`, but support depends on RN version,
platform, and New Architecture details. Older Android/elevation behavior does not match CSS
`box-shadow` cleanly.

Mitigation:

- decide the RN peer floor through Expo validation
- treat `box-shadow` as unsupported in Phase 0 unless Button needs it
- add a shadow policy before Card, Select, Toast, Dialog, or surfaces are marked supported
- avoid promising exact shadow parity across iOS, Android, and RN Web

### Risk: Expo Monorepo Setup Is Overconfigured

Modern Expo can auto-configure monorepos with `expo/metro-config`. Manual Metro resolver overrides
can create duplicate React/RN problems or stale config when Expo already supports the workspace.

Mitigation:

- start with Expo's default monorepo support
- only add custom Metro config after a reproduced bundling failure
- run package-manager diagnostics for duplicate `react` and `react-native`
- keep Expo playground dependency versions pinned after setup

### Risk: Accessibility Support Is Mistaken For Style Support

Even if styles compile, RN accessibility semantics for fields, modal surfaces, expandable controls,
combobox/select behavior, live regions, and described-by relationships differ from DOM semantics.

Mitigation:

- keep style compiler success separate from semantic support
- require native a11y contract tests per family
- require manual VoiceOver/TalkBack notes before marking form and overlay families supported
- avoid saying "same accessibility as web"

### Risk: Generated Output Becomes Too Hard To Review

If the generated IR is too noisy, reviewers will stop catching drift and unsupported behavior.

Mitigation:

- keep generated files deterministic and grouped by family
- emit compact inventories and diagnostics
- avoid timestamps
- add readable snapshots for Button before expanding

### Risk: Native Compiler Becomes A Browser CSS Reimplementation

Mitigation:

- publish a strict supported CSS subset
- fail closed on unsupported syntax
- keep component-specific escape hatches for hard cases
- never promise arbitrary CSS support

### Risk: Visual Parity Is Worse Than Handwritten Native Styles

Mitigation:

- compare generated styles against a small handwritten baseline in Phase 0
- allow generated styles plus native-only overrides
- keep the compiler output readable and auditable

### Risk: Accessibility Claims Become Too Broad

Mitigation:

- native package uses a separate support matrix
- field semantics require manual SR notes before marked stable
- docs avoid saying "same a11y as web" where RN cannot express it

### Risk: External Native Dependencies Hurt Adoption

Mitigation:

- keep v1 free of mandatory native modules
- use optional peers for icon/slider/date picker choices
- start with Expo validation

### Risk: Core Recipes Are Too DOM-Oriented

Mitigation:

- start with mapper layer in RN package
- add platform-neutral core metadata only when concrete duplication appears
- keep every core change useful to all adapters

## Open Decisions

1. Should React Native support be `@marwes-ui/react-native` or `@marwes-ui/native`?
   - Recommendation: `@marwes-ui/react-native` for clarity and ecosystem expectation.

2. Should the compiler live in `scripts/` or a package?
   - Recommendation: start in `scripts/`, promote to `packages/native-style-compiler` only if it
     becomes a reusable published artifact.

3. Should NativeWind be a dependency?
   - Recommendation: no for v1; run a spike only.

4. Should React Native Web be part of the support promise?
   - Recommendation: test it in Expo, but do not make it the primary target. Web already has
     `@marwes-ui/react`.

5. Should generated RN styles be committed?
   - Recommendation: yes, same as registry artifacts. Add `native-styles:check`.

6. Should `color-mix` remain in preset CSS?
   - Recommendation: yes for web, but compiler must either support the simple transparent mix case or
     introduce native-specific generated token helpers.

7. What is the first support label?
   - Recommendation: `experimental` until Button, Badge, Card, Text, Provider, and Input pass Expo
     app validation.

## Acceptance Criteria For A Good First PR

The first implementation PR should not try to ship the full package. It should prove the strategy.

Minimum acceptance criteria:

- `react-native-evaluation` has this plan.
- A follow-up branch can add the native design layer without changing existing web behavior.
- Compiler can parse one real preset CSS file and one native style manifest.
- Compiler output is deterministic and includes native design IR.
- Unsupported CSS fails with actionable diagnostics.
- Ignored CSS requires manifest rationale.
- No adapter imports preset CSS at runtime.
- No browser globals enter React Native package code.

## Bottom Line

React Native support is viable if Marwes treats it as a real fourth adapter with native constraints,
not a simple web adapter port.

The strongest path is:

1. preserve `@marwes-ui/core`
2. preserve `@marwes-ui/presets` as design source
3. add native style manifests only for information CSS cannot express safely
4. generate a strict native design IR from CSS plus manifests
5. materialize that IR through a typed React Native style resolver
6. ship `@marwes-ui/react-native` with an explicit support matrix
7. validate through Expo and native-focused tests

The native design layer is the key architectural bet. Build that first, prove it on Button, then
expand.
