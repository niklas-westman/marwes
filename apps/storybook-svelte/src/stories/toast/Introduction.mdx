# Toast Components

Marwes provides toast feedback at three layers: **purpose components** for common
status messages, **ToastContainer** plus **ToastProvider/useToast** for stacked
imperative delivery, and the raw **Toast** atom for custom layouts.

## Recommended: Purpose Components

Use purpose-built toasts when the message intent is already known:

```svelte
<script>
  import { ErrorToast, InfoToast, SuccessToast, WarningToast } from "@marwes-ui/svelte"
</script>

<SuccessToast>Project saved successfully.</SuccessToast>
<ErrorToast>Publishing failed. Please try again.</ErrorToast>
<WarningToast>Storage is almost full.</WarningToast>
<InfoToast>New release notes are available.</InfoToast>
```

## Base: ToastContainer + useToast (Molecule)

Use `ToastProvider` plus `useToast` when actions need to enqueue toasts imperatively.

```svelte
<script>
  import { ToastProvider } from "@marwes-ui/svelte"
</script>

<ToastProvider placement="top-right">
  <!-- Children can call useToast() to show toasts -->
  <slot />
</ToastProvider>
```

Inside a child component:

```svelte
<script>
  import { useToast } from "@marwes-ui/svelte"

  const toast = useToast()

  function handleSave() {
    toast.show({ message: "Project saved.", intent: "success" })
  }
</script>

<button type="button" onclick={handleSave}>Save project</button>
```

### ToastContainer Features

- **Stacking and placement** — supports top/bottom plus left/right viewport placement
- **Imperative workflow support** — `useToast()` exposes `show`, `dismiss`, and `clear`
- **Auto-dismiss** — provider-managed toasts close automatically unless `duration` is `null`
- **Accessible timing expectations** — ToastProvider defaults to short confirmation timing, pauses auto-dismiss while the toast is hovered or focused, and expects product teams to use `duration: null` or a longer explicit duration for critical or actionable messages
- **Purpose-aware rendering** — `intent` routes through the matching purpose toast variant

## Low-level: Toast (Atom)

Use the raw `Toast` atom when you need a bespoke layout or custom delivery mechanism.

```svelte
<script>
  import { Toast } from "@marwes-ui/svelte"
</script>

<Toast variant="rich" ariaLive="polite">
  Your changes have been saved.
</Toast>
```

The referenced Toast Figma frames use an inline `"Close"` text action inside the
toast surface. That action is not a standard `Button` or `CancelButton`; it is a
toast-local text treatment.

Pass a real interactive element when the toast needs a clickable inline action.
Use `mw-toast__action-button` for the toast-local underlined action treatment
when you need the adapter escape hatch.

```svelte
<Toast>
  {#snippet action()}
    <button type="button" class="mw-toast__action-button">
      Close
    </button>
  {/snippet}
  File uploaded successfully.
</Toast>
```

A literal string like `action="Close"` is visual inline text only.

## Component Reference

### Purpose Components

**`SuccessToast`** — Positive completion and confirmation feedback

**`ErrorToast`** — Failed actions and urgent alerts

**`WarningToast`** — Cautionary follow-up and risky state changes

**`InfoToast`** — Neutral updates and contextual notices

### Base Molecule

**`ToastContainer`** — Positioned toast viewport with stacking and dismiss handling

**`ToastProvider`** — Context provider that owns toast queue state

**`useToast`** — Imperative toast controller

### Atom

**`Toast`** — Low-level transient feedback surface
