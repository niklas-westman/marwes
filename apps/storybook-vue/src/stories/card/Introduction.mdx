# Card Components

Marwes provides cards at two layers: **purpose components** for common semantic
surfaces and the raw **Card** atom for custom composition.

## Component Architecture

```
Purpose Card
    ↓
Card (Atom)
```

## Recommended: Purpose Components

Use purpose-built cards when the surface has a clear product meaning:

```vue
<script setup lang="ts">
import { ProductCard, ProfileCard, StatCard } from "@marwes-ui/vue"
</script>

<template>
  <ProductCard>
    <template #title>Pro plan</template>
    Unlimited projects, advanced analytics, and priority support.
  </ProductCard>

  <ProfileCard>
    <template #title>Alex Morgan</template>
    Design systems engineer with ownership of component primitives.
  </ProfileCard>

  <StatCard>
    <template #title>Monthly recurring revenue</template>
    $128,400
  </StatCard>
</template>
```

## Base: Card (Atom)

Use the raw `Card` atom when you need a custom surface layout:

```vue
<script setup lang="ts">
import { Card } from "@marwes-ui/vue"
</script>

<template>
  <Card :data-attributes="{ 'data-surface': 'summary-card' }">
    <template #title>Project status</template>
    Build pipeline is healthy.
  </Card>
</template>
```

### Card Features

- **Optional title header** - renders consistent header spacing when a `title` slot is provided
- **Composition-friendly body** - accepts any Vue content through the default slot
- **Purpose-ready wrapper** - supports `dataAttributes` so semantic variants can layer metadata

## Accessibility notes

**Card is a passive `<div>`.** The card shell has no implicit role, heading, or interactive
semantics. Screen readers see it as a generic container. Meaning comes entirely from the
content placed inside.

**Card title slot renders as a styled `<span>`, not a heading.** Providing a `#title` slot
renders `<span class="mw-card__title">`. It does not update the document heading outline. If a
card surface needs to be findable by heading navigation, provide a real heading element
in the default slot rather than relying on the `#title` slot.

**Interactive card pattern.** If a card must be clickable, wrap its content in an `<a>` or
`<button>` rather than adding a click handler directly to the `Card` surface. A plain `<div>`
with a click handler is unreachable by keyboard users and invisible to assistive technology
as an interactive control.

```vue
<!-- ❌ Click handler on a passive div — keyboard inaccessible -->
<Card @click="handleClick">Project status</Card>

<!-- ✅ Anchor wraps the card content — keyboard and AT accessible -->
<a href="/projects/123">
  <Card>Project status</Card>
</a>
```

**`aria-*` and native div attrs pass through.** You can apply `aria-label`, `aria-labelledby`,
`role`, or other semantics directly to `Card` when product code needs to express richer meaning
for a specific composition.

## Component Reference

### Purpose Components

**`ProductCard`** - Product summaries and offer surfaces

**`ProfileCard`** - Identity and account summary surfaces

**`StatCard`** - KPI and snapshot metric surfaces
